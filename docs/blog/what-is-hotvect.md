What is Hotvect?
================

## Summary
hotvect is a framework for building ML-powered, real-time APIs that focuses on tasks like personalized ranking and top-k recommendation. Instead of isolating the ML logic from the business application, hotvect brings feature computation and decisioning logic directly into the JVM-based process that handles user requests, and integrates model inference efficiently. This approach breaks down the traditional separation between infrastructure and ML, enabling faster iteration cycles, seamless integration between algorithms and business logic, and mitigating common challenges like online-offline discrepancies. Broader parts of the ML application can be easily A/B tested. At the same time, hotvect uses Java’s dynamic loading to keep the different concerns loosely coupled. Hotvect is not a replacement for existing established ML frameworks. Instead, it is designed to work seamlessly with existing frameworks and libraries, such as feature stores, streaming feature calculation platforms, and other frameworks like SageMaker and Airflow, as well as ML libraries like TensorFlow, PyTorch, CatBoost, etc.

## The Challenges of Traditional Integrations
Traditionally, real-time recommendation systems are built with the following distinction:
The business application: Often implemented in statically typed languages like Java, C++, Scala, Go, or C#. It handles incoming requests, enriches the request before routing it to the recommendation engine. After the recommendation engine is done, it often applies various business logic before returning the finished response to the client.
The recommendation engine: Often implemented in Python and powered by frameworks like MLFlow and SageMaker. It provides the ML intelligence (the ML model).
In theory, this separation provides a clean architecture. The business layer sticks to infrastructure concerns while the recommendation engine handles the ML. Often these two components are also maintained by different teams (the platform team and the ML team). This allows the recommendation engine to be improved and A/B tested without touching the business application, thereby reducing coupling and increasing iteration speed.
However, in practice, this separation of concern is often violated. The business application often performs operations such as projection, truncation, and aggregation of input data before passing them onto the recommendation engine. I.e. it is performing feature engineering. This "leaking" of feature engineering into the business application has severe drawbacks.
When logic for feature engineering is placed inside the business application, it usually operates outside the scope of the experimentation framework, making A/B testing of this logic difficult. It also necessitates coordination between components and teams, reducing iteration speed. Testing is also usually difficult, leading to issues that are hard to debug, such as subtle online-offline discrepancies. As a result, it's not uncommon to see teams focus on what they can change in the recommendation engine, and avoid touching any logic that was implemented in the business application—even when there is a low-hanging fruit.
The reason why this "leaking" is so common is this: to put all of the feature engineering code into the python component, the raw data must be sent to it. This increases the size of the payload, adding latency and overhead. Secondly, Python's lack of robust concurrency support and lower efficiency make it difficult to perform these calculations while meeting the required system performance requirements.

## The Response of the Industry to this Problem
Recent industry research and experiences from companies like LinkedIn, DoorDash, Airbnb, Instacart, and others have highlighted the importance of feature platforms in tackling this issue. These platforms handle feature computation, storage, and retrieval, and enable data scientists to iterate fast on features. The type of feature calculation can be categorized as follows:
Batch features (BF): Precomputed periodically, sometimes incorporated into the model itself. Expected lag ranges between 1 to 24 hrs.
Near real-time features (NRTF): Precomputed using streaming engines like Flink or Spark Streaming. Expected lag ranges from several minutes to several seconds.
Real-time features (RTF): Computed on-demand at prediction time. The most important distinction is that it can use information only available in the request itself.
Evidence has shown that moving from batch-only pipelines to NRT and RT feature computation significantly boosts the quality of recommendations. However, it also brings significant complexity and challenges.
BF and NRTF have received extensive support investment, and have good off-the-shelf software that support them. For example, feature stores like Hopsworks and feature platforms like Feathr support them. However, support for RTF remains scarce. Frameworks like Hopsworks and MLFlow support real-time feature transformation by allowing developers to register Python User Defined Functions (UDFs). However, as described above, performing this transformation using Python is challenging.
Frameworks like Facebook's F3 work by compiling transformations defined in a Python-like Domain Specific Language (DSL) into efficient low-level implementations (such as optimized C++ code). Other examples include Uber’s Michelangelo Palette and Airbnb’s Chronon, which similarly unify feature definitions to ensure that training and serving pipelines run identical logic. Similarly, LinkedIn’s Feathr and DoorDash’s Fabricator adopt a declarative approach that enables a single source of truth for feature transformations, ensuring consistent, low-latency feature computation across batch and streaming environments.

## What hotvect Does Differently
As mentioned above, many frameworks focus on providing a declarative interface through DSLs that are then converted into more efficient instructions by the framework. Sometimes the same feature declaration translates to different implementations depending on whether the calculation happens in an online context or offline context.
Hotvect solves this problem differently. The transformations are directly defined in Java (or any JVM compatible language) as plain Java code, and the resulting jar package is shipped to the online and offline "containers." Containers in this context are runtime environments like Spring Boot, Apache Spark, or command-line Java applications.
By using plain Java, it offers the following benefits over the more common DSL approach:
Flexibility: Avoids the constraints of a DSL by giving full access to Java’s capabilities, enabling more complex and customized transformations.
Ease of Maintenance: Developers can leverage existing Java tooling and libraries, simplifying debugging, testing, and optimization.
Broader scope: By sharing the language with the containing application, a much broader scope of operations (such as fetching of features, logging) can be encompassed into the scope, which makes A/B tests of larger changes easier.
Consistency Across Environments: The same Java code can be executed in online systems (e.g., Spring Boot) and offline systems (e.g., Apache Spark), ensuring even stronger consistencies.
Debuggability: Using Java improves the ability to debug and trace feature transformations compared to Python-based DSL approaches. This ensures that issues can be identified and resolved more efficiently in both online and offline environments.
While hotvect offers significant advantages, it also has drawbacks.
Steeper Learning Curve: Data Scientists are typically unfamiliar with JVM-based languages and may face challenges working effectively in it.
Weaker support for non-JVM libraries during feature computation: Python UDFs can make use of python feature computation libraries like nltk, but this is much harder when the feature computation runs in Java
Risk of Coupling: While hotvect enables tighter integration between ML and business logic, there is a risk of creating overly coupled systems that may be harder to refactor or scale independently.
Resource Management Complexity: Running more logic (i.e. business logic and feature computation, and even inference in some cases) in the same process can make resource management harder, making it potentially less efficient.

## Synergy with Existing ML Infrastructures
hotvect integrates with existing ML infrastructures rather than replacing them. It is designed to be used alongside frameworks like SageMaker, Airflow, Hopsworks, and Databricks, as well as libraries like TensorFlow, PyTorch, CatBoost, and XGBoost etc. It does not replace feature platforms for batch (BF) or near real-time (NRTF) calculations, nor does it replace computation frameworks like Spark, Flink, or Apache Beam.
Hotvect addresses real-time features (RTF) as well as integration with feature stores and other databases, as well as model serving and A/B testing. It also addresses business logic (e.g. heuristics), and integration between vector databases, document databases and recommendation models. This holistic approach bridges the gap between feature computation, business decisioning, and ML inference.
Hotvect allows co-location of business logic, feature computation, and ML inference within a single component (while the ML inference can be  externalized if needed). This design enables more complex interactions between algorithms and business logic. For example, the retrieval of candidates from vector or document databases, combined with feature computation and model inference, can be executed seamlessly, providing greater flexibility in designing algorithms that integrate diverse components. Existing technologies like PyTorchserve and MLFlow have features to combine different models, but hotvect extends this capability to databases and other business logic. Using Java here gives more latency budget for performing these complex interactions, too.

## Conclusion
Hotvect integrates feature computation, model inference, and business logic within a JVM-based runtime environment. By doing so, it offers a streamlined approach for deploying and iterating on real-time recommendation systems, reducing the complexity and overhead often associated with the traditional separation of application and ML layers. Its reliance on Java code for transformations eliminates the need for DSLs and ensures stronger consistency between online and offline environments. Though it introduces new challenges—such as a learning curve for data scientists unfamiliar with JVM-based workflows—it provides a valuable alternative for organizations seeking tighter integration, improved debuggability, and increased iteration speed. Rather than displacing existing ML infrastructures, hotvect enhances them, enabling data science and engineering teams to deliver more effective, customizable, and resilient real-time recommendation solutions.
